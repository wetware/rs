# Economic Agent Platform Vision

Generated by CEO review on 2026-03-18.

## Thesis

Wetware is not a "P2P OS" — it is the **economic coordination layer for
autonomous agents.** Agents that hold assets, trade services, coordinate
across trust boundaries, and attest to their behavior — with
cryptographic guarantees about what they can and cannot do.

Every major AI agent framework (Langchain, CrewAI, Eliza, AutoGPT) has
the same Achilles' heel: agents have **ambient authority.** They run as
the user. Wetware's capability model makes it impossible for agents to
exceed their authority. That is the moat.

## Design Thesis: E-lang + Clojure + Unix

The shell is the primary UX surface for capability-based thinking.

- **E-lang:** Capabilities are first-class values — bind, pass, compose,
  attenuate, inspect.
- **Clojure:** Syntax, data model, immutable values, composability.
- **Unix:** Process model (PATH, stdin/stdout, pipes), init.d service
  management.

### Three-Layer Stdlib

```
┌────────────────────────────────────────────────────────────────┐
│  LAYER 3: std/ WASM agents  (like /usr/bin)                    │
│  Heavy data transformation operations.                         │
│  Called: PATH lookup → executor.runBytes → child process        │
├────────────────────────────────────────────────────────────────┤
│  LAYER 2: Capability verbs  (shell built-ins)                  │
│  discover, grant, attenuate, connect, inspect                  │
│  Called: recognized by evaluator → RPC call                    │
│  THIS IS WHERE THE E-LANG THINKING LIVES                       │
├────────────────────────────────────────────────────────────────┤
│  LAYER 1: Glia core  (like clojure.core)                       │
│  def, let, fn, if, do, map, filter, ->, ->>, comp              │
│  Called: interpreted directly by Glia evaluator                 │
└────────────────────────────────────────────────────────────────┘
```

Resolution: everything is a function call. Strategy (local eval / RPC /
process spawn) is an implementation detail. "Everything is a capability
invocation" = Plan 9's "everything is a file."

## Five Pillars (12-Month Vision)

```
┌──────────────────────────────────────────────────────────────────┐
│                     WETWARE PLATFORM                              │
├──────────┬──────────┬───────────┬───────────┬───────────────────┤
│ 1.WALLET │ 2.MARKET │ 3.VERIFY  │ 4.GOVERN  │ 5.TOOLING        │
│ Agent as │ Agent-to │ Content-  │ Membrane  │ Ship fast,        │
│ economic │ -agent   │ addressed │ policy    │ debug easy         │
│ actor    │ exchange │ WASM+TEE  │ language  │                    │
└──────────┴──────────┴───────────┴───────────┴───────────────────┘
```

**Pillar 1 — Wallet:** Agents hold assets. Host-side signing (key never
in WASM). Membrane attenuation: spending limits, whitelists.

**Pillar 2 — Market:** Capability Exchange Protocol. Agents exchange
capabilities across trust boundaries using payment, reciprocity,
reputation, governance, and TEE attestation.

**Pillar 3 — Verify:** Content-addressed WASM + TEE attestation.
Prove what code runs, where it runs, and what authority it has.

**Pillar 4 — Govern:** Reference attenuating Membranes. Composable
GraftBuilder chain. Policy enforcement at the capability level.

**Pillar 5 — Tooling:** Developer experience competitive with Fly.io.
`ww publish`, hot reload via epoch, process visibility, debugging.

## Accepted 3-Month Scope

Prioritized sequencing:

1. **Service Discovery Protocol** — DHT-based agent discovery
2. **Glia stdlib** — incremental: `def/let/fn/if/do` → threading macros
   → destructuring
3. **Init.d refactor** — parallel eval, error recovery, dispatch table
   (issue #161)
4. **Developer tooling** — workflow design first, then commands
5. **Agent Wallet** — host-side signing, Membrane-attenuated spending
6. **Hot code reload** — build + publish + epoch advance
7. **Reference Attenuating Membranes** — 3-4 reference implementations
8. **Identity-based authority attenuation** — policy function mapping
   VerifyingKey → capability set (different peers get different authorities)
9. **Wire up remote peer auth** — guest-exported Membrane served to
   incoming libp2p connections (prerequisite for service discovery)
10. **Multi-agent demo** — price oracle network
11. **TEE research spike** — which TEE, Wasmtime integration story
12. **Membrane auth tests** — integration test coverage

## Open Design Questions

- Stdlib layer boundaries: what belongs in core vs verbs vs WASM agents?
- Dev tooling: specific workflows before CLI commands
- Capability Exchange Protocol: concrete protocol spec TBD
- TEE: TDX vs SEV-SNP? Wasmtime-in-enclave integration story?
- Admin/metrics: separate interface design (not public libp2p streams)
- Atom-level authority attenuation: different identities → different
  starting capability sets. Currently HostGraftBuilder gives every
  grafter the same full capability set. Need: policy function mapping
  VerifyingKey → GraftBuilder (different keys get different capabilities).
  This is the core Membrane governance mechanism.
- pid0 should authenticate to the host Membrane the same way remote
  peers do — no special "skip auth" path. The host maps pid0's
  identity to a capability set via GraftPolicy. Same code path for
  local and remote grafters. Operator controls pid0's authority via
  config/policy, not via code. This also makes the TEE story
  consistent (pid0 attestation verified same as remote peers).
- Service discovery trust model: name-based (usable, poisonable) vs
  CID-based (tamper-proof, needs registry). Start with names, add CID
  verification when TEE lands.
